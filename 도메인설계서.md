## 1. 프로젝트 개요

### 1.1 프로젝트 목적

취업·이직 준비자가 **면접 일정·예상 질문·답변 초안**을 한곳에서 관리하고, 면접 전 **준비 질문 풀**과 **개인 질문 뱅크**를 활용해 반복 준비를 체계화한다. 또한 LLM(Groq)을 이용해 **연습 답변에 대한 STAR 중심 첨삭**을 제공하여, 답변 품질을 스스로 점검·개선할 수 있게 한다.

### 1.2 프로젝트 범위

**포함 범위:**

- 사용자 계정: 회원가입, 로그인, 본인 정보 조회
- 면접(Interview): 회사·직무·일정·차수(메모) 단위 관리 및 CRUD
- 면접 질문(InterviewQuestion): 특정 면접에 속한 질문 목록, 출처(준비 질문/뱅크/직접 입력), 순서, 답변·복기 텍스트
- 준비 질문(PreparationQuestion): 면접 일정과 무관하게 쌓아 두는 질문·연습 답변, 뱅크 연동 또는 직접 입력
- 질문 뱅크(QuestionBankQuestion): 사용자별 재사용 가능한 질문 자산(수동 등록·면접에서 가져오기 등), 보관(archived) 처리
- AI 연습 답변 코칭: 준비 질문에 대한 초안 텍스트를 입력하면 STAR 관점 피드백 생성(외부 API 키 설정 시)

**제외 범위:**

- OAuth·소셜 로그인, 이메일 인증, 비밀번호 찾기 등 고급 인증 플로우(현재는 username/password + 클라이언트에 사용자 ID 보관 방식)
- 역할 기반 관리자 콘솔(현재 `UserRole`은 `USER` 단일)
- 실시간 화상 면접·캘린더 외부 연동(Google Calendar 등)
- 모바일 네이티브 앱(웹 클라이언트 기준)

### 1.3 주요 이해관계자 (Stakeholders)

| 구분 | 역할 | 주요 관심사 |
|------|------|-------------|
| 구직자(일반 사용자) | 시스템 사용자 | 면접별 질문 정리, 준비 노트 축적, 답변 첨삭으로 완성도 향상 |
| 제품/기획 | 요구사항 정의 | 출처 타입·스냅샷 정책, 권한 모델 확장 여지 |
| 개발팀 | 구현·운영 | 도메인 일관성, 멀티테넌트(사용자별 데이터 격리), AI 장애 시 graceful degradation |
| 인프라/보안 | 운영 | 비밀번호 해시 저장, API 키 관리, CORS·헤더 기반 식별의 한계 인지 |

## 2. 비즈니스 도메인 분석

### 2.1 핵심 비즈니스 프로세스

#### 2.1.1 회원 가입·로그인 프로세스

**가입**

1. `username`(3-30자, 영문·숫자·`_`만) · `password`(8-100자) 입력
2. 시스템이 `username` 중복 여부 확인 → 중복 시 거절
3. 비밀번호는 단방향 해시로 저장, 역할 `USER`, 상태 `ACTIVE`로 사용자 생성

**로그인**

1. `username`으로 사용자 조회
2. 상태가 `ACTIVE`이고 비밀번호가 일치하면 성공, 응답에 사용자 식별 정보 반환
3. 이후 API 호출 시 클라이언트가 `X-User-Id` 헤더로 본인 식별(서버는 해당 ID 소유 리소스만 허용)

#### 2.1.2 면접·질문 준비 프로세스

1. 사용자가 면접을 생성(회사명, 직무명, 면접일, 차수/메모 등)
2. 면접에 질문을 추가 — 출처 유형에 따라:
   - **준비 질문에서 가져오기**: 본인 소유 준비 질문 참조 + 질문 본문 스냅샷 저장
   - **질문 뱅크에서 가져오기**: 본인 소유 뱅크 질문 참조 + 스냅샷
   - **직접 입력**: 스냅샷 텍스트만으로 생성
3. 질문 순서 조정(`sortOrder`), 필요 시 답변(`answerText`)·복기(`reviewText`) 기록
4. 면접 목록은 기본적으로 면접일 기준 내림차순 페이지네이션

#### 2.1.3 준비 질문·질문 뱅크 운용 프로세스

1. **질문 뱅크**: 자주 쓰는 질문을 등록·수정·보관(archive). 보관된 자산은 본문 수정 불가(정책상).
2. **준비 질문**: 뱅크에서 복사하거나 직접 문장을 넣고, 연습 답변(`practiceAnswer`)을 옵션으로 유지
3. 준비 질문 삭제 시, 이를 참조하던 면접 질문은 FK 제약을 피하기 위해 **출처를 CUSTOM으로 바꾸고 스냅샷은 유지**(준비 질문 링크만 해제)

#### 2.1.4 AI 연습 답변 코칭 프로세스

1. 사용자가 특정 준비 질문에 대해 연습 답변 초안을 제출
2. 서버가 Groq(OpenAI 호환) API로 첨삭 요청
3. API 키 미설정 시 “AI 미구성” 오류로 기능만 비활성, 나머지 도메인 기능은 정상 동작

**비즈니스 규칙(요약):**

- 모든 비즈니스 데이터는 **소유자(사용자) 단위로 격리**; 타인 리소스 ID를 알아도 `ACCESS_DENIED` 또는 NOT_FOUND 정책에 따름
- `InterviewQuestion.sortOrder`는 0 이상; 재정렬 시 목록은 **중복 없이 전체 ID를 한 번에** 보내야 함
- 엔티티 내부 검증: 답변/복기/연습답변/질문 스냅샷 갱신 시 빈 문자열 불가(클리어는 별도 경로)

### 2.2 비즈니스 이벤트

| 이벤트 | 트리거 | 결과 |
|--------|--------|------|
| 회원가입 완료 | 중복 없는 username으로 등록 성공 | `User` 영속화, 기본 역할·상태 부여 |
| 로그인 성공 | ACTIVE 사용자 + 비밀번호 일치 | 클라이언트가 이후 `X-User-Id`로 API 접근 |
| 면접 생성 | 유효한 입력으로 POST | `Interview` 생성, 상태 `DRAFT`(DB 호환용, API 비노출) |
| 면접 질문 추가 | 출처 타입과 ID/텍스트 조합 유효 | `InterviewQuestion` 생성 및 면접에 연결 |
| 면접 질문 순서 변경 | 재정렬 API에 전체 질문 ID 목록 | 각 질문 `sortOrder` 재부여 |
| 준비 질문 삭제 | 소유자 요청 | 참조 중인 면접 질문은 CUSTOM 전환 후 준비 질문 삭제 |
| 질문 뱅크 보관 | archived 플래그 설정 | 해당 자산 본문 수정 불가 |
| AI 코칭 요청 | 준비 질문 + 초안 텍스트 | 성공 시 첨삭 텍스트 반환; 키 없음/외부 오류 시 정해진 오류 코드 |

## 3. 핵심 도메인 객체 (Domain Objects)

### 3.1 도메인 객체 식별 매트릭스

| 도메인 객체 | 유형 | 중요도 | 비고 |
|-------------|------|--------|------|
| User | Entity | 높음 | 모든 리소스의 소유 주체 |
| Interview | Entity | 높음 | 면접 단위 애그리게이트 루트(자식: InterviewQuestion) |
| InterviewQuestion | Entity | 높음 | 면접별 질문 + 스냅샷·출처 |
| PreparationQuestion | Entity | 높음 | 면접 전 준비용 질문 풀 |
| QuestionBankQuestion | Entity | 중간 | 재사용 질문 자산 |
| InterviewStatus | Value Object(Enum) | 중간 | DRAFT, COMPLETED — DB용, API 미노출 |
| InterviewQuestionSourceType | Value Object(Enum) | 중간 | FROM_PREPARATION, FROM_BANK, CUSTOM |
| PreparationQuestionSourceType | Value Object(Enum) | 중간 | FROM_BANK, CUSTOM |
| QuestionAssetSourceType | Value Object(Enum) | 낮음 | MANUAL, IMPORTED_FROM_INTERVIEW |
| UserStatus / UserRole | Value Object(Enum) | 중간 | 상태 ACTIVE/INACTIVE, 역할 현재 USER만 |
| (감사) createdAt/updatedAt | Value Object | 낮음 | `BaseEntity`에서 관리 |

### 3.2 상세 도메인 객체 정의

#### 3.2.1 User (사용자)

**역할:** 시스템에 가입한 구직자 계정. 모든 면접·질문 데이터의 논리적 소유자.

**주요 속성:**

| 속성 | 설명 |
|------|------|
| id | 대리키(자동 증가) |
| username | 로그인 ID, DB 유니크, 최대 30자 |
| passwordHash | BCrypt 등으로 인코딩된 비밀번호 |
| role | 현재 구현은 `USER`만 사용 |
| status | `ACTIVE` / `INACTIVE` |
| createdAt, updatedAt | `BaseEntity` |

**주요 행동:**

- `register(username, passwordHash)`: 신규 사용자 팩토리
- `passwordMatches(encoder, rawPassword)`: 로그인 검증
- `deactivate()`: 비활성 전환(멱등)

**비즈니스 규칙:**

- username은 DB에서 유일해야 함
- INACTIVE 사용자는 로그인 불가(자격 증명 오류와 동일 응답으로 처리 가능)

---

#### 3.2.2 Interview (면접)

**역할:** 특정 회사·직무·일정 단위의 면접 세션. 소속 질문은 컬렉션으로 관리(cascade, orphanRemoval).

**주요 속성:**

| 속성 | 설명 |
|------|------|
| id | 대리키 |
| user | 소유자(Many-to-One, 필수) |
| companyName | 회사명, 최대 120자 |
| positionTitle | 직무명, 최대 120자 |
| interviewDate | 면접일(LocalDate) |
| status | `DRAFT` / `COMPLETED` — 스키마 호환용 |
| interviewRound | 차수 설명(DB 컬럼명 `memo`), 최대 2000자 |
| interviewQuestions | 질문 목록(One-to-Many) |

**주요 행동:**

- `create(...)`: 생성 시 status = DRAFT
- `update(...)`: 메타데이터 수정
- `addInterviewQuestion(q)`: 질문 추가 및 sortOrder 기준 정렬

**비즈니스 규칙:**

- user는 필수
- 조회·수정·삭제는 **소유 user와 요청 사용자 일치** 필요(애플리케이션 서비스 계층)

---

#### 3.2.3 InterviewQuestion (면접 질문)

**역할:** 한 면접에 속한 개별 질문. 출처에 따라 준비 질문·뱅크 질문을 참조할 수 있으나, 본문은 `questionTextSnapshot`으로 보존.

**주요 속성:**

| 속성 | 설명 |
|------|------|
| id | 대리키 |
| interview | 소속 면접(필수) |
| preparationQuestion | 준비 질문 출처(선택) |
| questionBankQuestion | 뱅크 출처(선택) |
| sourceType | FROM_PREPARATION / FROM_BANK / CUSTOM |
| questionTextSnapshot | 질문 본문 스냅샷, 최대 4000자 |
| sortOrder | 표시 순서, 0 이상 |
| answerText | 답변 초안, 최대 8000자(옵션) |
| reviewText | 면접 후 복기, 최대 8000자(옵션) |

**주요 행동:**

- `fromPreparation` / `fromBank` / `fromCustom`: 출처별 팩토리
- `updateAnswerText` / `clearAnswerText`, `updateReviewText` / `clearReviewText`
- `updateSortOrder`: 0 미만 불가
- `unlinkPreparationForDeletion`: 준비 질문 삭제 시 참조 해제 및 CUSTOM 전환

**비즈니스 규칙:**

- `sourceType`과 FK 조합은 서비스 계층에서 일관되게 검증(FROM_PREPARATION이면 preparationQuestionId 필수 등)
- 답변·복기 본문 업데이트 시 빈 문자열 불가(엔티티 레벨)

---

#### 3.2.4 PreparationQuestion (준비 질문)

**역할:** 면접 카드에 올리기 전 “연습 장”에 쌓는 질문. 뱅크에서 왔는지 직접 썼는지 `sourceType`으로 구분.

**주요 속성:**

| 속성 | 설명 |
|------|------|
| id | 대리키 |
| owner | 소유 User(필수) |
| questionBankQuestion | 뱅크 참조(선택) |
| sourceType | FROM_BANK / CUSTOM |
| questionTextSnapshot | 본문, 최대 4000자 |
| practiceAnswer | 연습 답변, 최대 8000자(옵션) |

**주요 행동:**

- `fromBank` / `fromCustom`
- `updateQuestionTextSnapshot`, `updatePracticeAnswer` / `clearPracticeAnswer`

**비즈니스 규칙:**

- 스냅샷·연습답변 갱신 시 빈 문자열 불가(클리어는 전용 메서드)
- 삭제 시 연결된 면접 질문은 스냅샷 유지 + CUSTOM 전환

---

#### 3.2.5 QuestionBankQuestion (질문 뱅크 항목)

**역할:** 사용자별로 재사용하는 질문 자산. 보관 시 편집 제한.

**주요 속성:**

| 속성 | 설명 |
|------|------|
| id | 대리키 |
| owner | 소유 User(필수) |
| questionText | 질문 본문, 최대 4000자 |
| sourceType | MANUAL / IMPORTED_FROM_INTERVIEW |
| archived | 보관 여부 |

**주요 행동:**

- `create`, `archive`, `setArchived`
- `updateQuestionText`: archived이면 예외(API에서 ARCHIVED_QUESTION_NOT_EDITABLE)

**비즈니스 규칙:**

- 보관된 자산은 본문 수정 불가
- 소유자가 아닌 접근은 서비스에서 거절

## 4. 도메인 관계도

### 4.1 개념적 관계도

```
[User] 1 ────── N [Interview]
[User] 1 ────── N [PreparationQuestion]
[User] 1 ────── N [QuestionBankQuestion]

[Interview] 1 ────── N [InterviewQuestion]

[PreparationQuestion] 0..1 ─── N [InterviewQuestion]   (FROM_PREPARATION; 삭제 시 링크 해제)
[QuestionBankQuestion] 0..1 ─── N [InterviewQuestion]  (FROM_BANK)
[QuestionBankQuestion] 0..1 ─── N [PreparationQuestion] (FROM_BANK)
```

### 4.2 관계 상세 설명

| 관계 | 카디널리티 | 설명 | 제약조건 |
|------|------------|------|----------|
| User ↔ Interview | 1:N | 한 사용자는 여러 면접 보유 | 면접은 항상 소유자 존재 |
| User ↔ PreparationQuestion | 1:N | 준비 질문은 사용자별 격리 | owner 필수 |
| User ↔ QuestionBankQuestion | 1:N | 질문 뱅크는 사용자별 격리 | owner 필수 |
| Interview ↔ InterviewQuestion | 1:N | 면접 삭제 시 질문 cascade | sortOrder로 순서 표현 |
| PreparationQuestion ↔ InterviewQuestion | 1:N (옵션) | 준비에서 끌어온 면접 질문만 연결 | 준비 질문 삭제 시 FK 해제 + CUSTOM |
| QuestionBankQuestion ↔ InterviewQuestion | 1:N (옵션) | 뱅크에서 끌어온 경우 연결 | — |
| QuestionBankQuestion ↔ PreparationQuestion | 1:N (옵션) | 뱅크 기반 준비 질문 | — |

## 5. 비즈니스 규칙 (Business Rules)

### 5.1 사용자·인증 관련

| 규칙 ID | 규칙 내용 | 우선순위 |
|---------|-----------|----------|
| BR-U001 | username은 3~30자, 영문·숫자·언더스코어만 허용 | 높음 |
| BR-U002 | username은 시스템 전역 유일 | 높음 |
| BR-U003 | password는 8~100자(평문 검증은 DTO) | 높음 |
| BR-U004 | INACTIVE 사용자는 로그인 불가 | 높음 |
| BR-U005 | 보호 API는 `X-User-Id` 필수·숫자 형식 | 높음 |
| BR-U006 | 리소스 접근 시 소유자 ID와 `X-User-Id` 일치 | 높음 |

## 6. 용어 정의 (Glossary)

### 6.1 비즈니스 용어

| 용어 | 정의 | 영문 | 비고 |
|------|------|------|------|
| 면접 | 회사·직무·일정(및 차수 메모)으로 구분되는 한 번의 면접 단위 | Interview | 질문 목록을 자식으로 둠 |
| 면접 질문 | 특정 면접에 매핑된 질문 한 건(스냅샷 포함) | Interview Question | 출처는 준비/뱅크/직접입력 |
| 준비 질문 | 면접 카드에 올리기 전에 모아 두는 질문·연습답변 | Preparation Question | AI 코칭의 주 대상 |
| 질문 뱅크 | 사용자가 재사용하기 위해 보관하는 질문 자산 | Question Bank | 보관 시 편집 잠금 |
| 질문 스냅샷 | 참조 시점의 질문 본문 복사본 | Question Text Snapshot | 원본 변경·삭제와 분리 |
| 보관 | 질문 뱅크 항목을 수정 불가 상태로 두는 플래그 | Archived | archive 해제 정책은 제품 결정 |
| STAR 코칭 | Situation-Task-Action-Result 관점의 답변 구조 피드백 | STAR Coaching | LLM 응답 품질은 모델·프롬프트 의존 |
